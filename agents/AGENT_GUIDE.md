# Agent Guide

## 1. 项目在做什么

本项目不是通用摘要器，也不是临床/法律/投资判断系统。它的核心目标是：

1. 把一个人物的大量公开材料整理成结构化证据。
2. 让多个独立理论视角（lens）在相同证据层上并行分析。
3. 通过批判审计、跨视角综合与报告生成，得到一个“多镜头人物画像”。

系统当前的基本流程是：

1. `evidence_assembler`：从原始 source 生成 `timeline` 和 `evidence_cards`
2. `discipline` agents：每个 lens 独立分析同一批证据
3. `critic`：检查理论越界、证据不足、过度推断
4. `synthesizer`：做跨 lens 的收敛、互补、张力和场景推断
5. `reporter`：把结构化结果转成可读报告

换句话说，项目的“真货”不是漂亮文风，而是：

- 证据层是否干净
- lens 是否彼此独立
- 分析是否真正受理论框架约束
- 综合是否保留冲突，而不是硬捏成统一结论

## 2. 数据观：先证据，再解释

本项目采用的是“证据先行”而不是“直觉先行”的写法。

原始 source 不直接喂给综合报告，而是先被拆成两类结构化对象：

- `timeline`
  只回答“发生了什么”
- `evidence_cards`
  只保留可引用、可回溯的证据片段

因此，后续所有 discipline/lens 的分析都应建立在 `evidence_cards` 上，而不是把原始材料重新想象一遍。

基本规则：

- 结论优先引用 `ev_XXX`
- 时间相关判断可以补充参考 `evt_XXX`
- 不要把第三方摘要硬写成当事人自述
- 不要把“看起来合理”当成“证据已支持”

## 3. shared_constructs 的作用

`shared_constructs` 是全项目的公共坐标系。

它的作用不是强迫所有学科说一样的话，而是让不同理论框架都能落到一组可比较的维度上，例如：

- `agency_orientation`
- `epistemic_style`
- `control_orientation`
- `risk_posture`
- `metacognition`

这套 construct 的意义是：

- 为不同 lens 提供共同输出接口
- 为 `critic` 和 `synthesizer` 提供统一比较平面
- 避免每个学科都输出一套无法对接的私有概念

写 skill 时应记住：

- construct 是“翻译接口”，不是理论本体
- 同一个 construct 可以被不同 lens 用不同理论语言解释
- 如果某个 construct 在该框架下确实不适用，应写 `not_applicable`
- 不要为了“填满所有格子”而硬凑分析

## 4. lens 的定义：一个理论镜头，不是一个学科总论

项目里的最小分析单位不是 discipline，而是 lens。

一个好的 lens 应满足 4 个条件：

1. 有清晰的理论来源或方法论
2. 有自己独立的“第一问题”
3. 能从公开材料里抓到可观察证据
4. 能映射到 shared constructs

### 什么叫“独立的第一问题”

如果两个 lens 最后只是换术语说同一件事，那就不是两个 lens。

每个 lens 都应该能用一句话定义自己的主问题，例如：

- “这个人如何对待睡眠、恢复和节律？”
- “这个人是否把高压激活当成默认运行模式？”
- “这个人的工作方式如何制造负荷与安全边界问题？”

这是区分 lens 的最重要规则。

### 允许交叉，但不允许重复

项目允许不同 lens 围绕同一批证据形成交叉支持。

例如：
- 一个 lens 从认知控制解释同一行为
- 另一个 lens 从职业疲劳解释同一行为

这是收敛，不是问题。

真正的问题是：

- 主问题相同
- 优先证据相同
- 结论只有措辞差异，没有解释差异

因此，新增 lens 时最好明确写出：

- 唯一主问题
- 优先证据
- 常见混淆
- 不应越界

## 5. skill 文档应如何写

一个正式 skill 文档最好包含以下结构：

1. 核心理论框架
2. 锚点人物与流派
3. 分析方法论
4. Construct 分析指南
5. 典型发现模式

必要时可额外加入：

- lens 定位与去重边界
- 不应越界
- 适合抓取的证据类型

skill 文档的本质不是“知识介绍”，而是“约束分析输出的规范”。

因此一个好的 skill 应该做到：

- 告诉模型应使用哪些概念
- 告诉模型不要使用哪些概念
- 告诉模型优先看什么证据
- 告诉模型哪些推断是不该做的

## 6. prompt 设计原则

discipline prompt 的职责，不是提供领域知识本身，而是提供：

- 输出 schema
- 证据使用规则
- 领域边界
- 理论约束

推荐原则：

- 明确要求“严格使用上述理论框架的概念和术语”
- 明确要求“引用具体 evidence card ID”
- 明确要求“证据不足时用 weak / not_applicable”
- 明确要求“不要越界做诊断/因果断言/病理标签”

高风险领域要额外更硬：

- 医学：不做临床诊断，不推断未公开病史
- 心理学：不把行为模式直接升级为精神疾病结论
- 法律/金融/医学等：优先讨论模式、风险、边界，不直接做确定性专业判断

## 7. source type 的使用原则

不是所有 source 的证据强度都一样。

项目里已有 `skills/source_context/*.yaml` 来区分：

- 新闻
- 推文
- 演讲
- 访谈
- 传记
- 法律文件
- 白皮书

一般原则：

- `self_statement`：适合判断自述立场、公开价值观
- `direct_report`：适合判断行为与事件，但仍要防止记者或叙述者框架偏差
- `third_party_summary`：适合做弱支持或背景补充，不适合做强心理/医学结论

写 lens 时应优先：

- 先看直接行为和自述
- 再看他人归因
- 最后才看高度解释性的二手概括

## 8. emergent_constructs 的使用原则

`emergent_constructs` 是这个项目最有创造性的部分，但也最容易滥用。

只有在以下条件同时满足时，才应该新增 emergent construct：

1. 该理论框架确实看到了 shared constructs 之外的重要维度
2. 该维度不是换个名字重复已有 construct
3. 该维度有具体证据支持
4. 该维度能说清“为什么 shared_constructs 没覆盖它”

建议控制为：

- 每个 lens 0-2 个

常见错误：

- 把普通总结句包装成 emergent construct
- 用花哨名字重命名已有 construct
- 没有证据，只靠理论脑补

## 9. 这个项目更像“结构化娱乐”，但底层仍要严谨

从传播角度看，这个项目可以往“多角度人格分析”“类似 MBTI / 星座 / 血型 的可读体验”靠近。

但这只应该发生在表达层，而不应该污染证据层和分析层。

建议区分两层：

### 底层分析层

- 严格证据约束
- 理论边界清楚
- 不越界做诊断
- 允许 `weak` / `not_applicable`
- 保留冲突与不确定性

### 上层表达层

- 可读性更强
- 术语可稍微转译
- 风格可更传播友好
- 结构可更“画像化”

原则是：

> 可以把结果写得更像一个好读的人物分析，但不能把分析过程写成神棍判断。

## 10. 对 agent 的通用行为要求

任何 agent 在本项目里都应遵守以下行为准则：

- 不要越出自己的理论边界
- 不要替别的 lens 发言
- 不要把第三方话术当成事实
- 不要因为结论“看起来合理”就省略证据
- 不要为了完整性硬填所有 construct
- 不要消解矛盾；矛盾本身是重要发现
- 不要把规范判断伪装成事实判断

如果证据不足，最好的输出不是“更聪明的猜测”，而是：

- `weak`
- `not_applicable`
- 明确指出边界

## 11. 新增一个 discipline 时的推荐流程

1. 先定义这个 discipline 在项目里的边界
2. 明确它与相邻学科的分工
3. 选 4-8 个真正有区分度的 lens
4. 为每个 lens 写 skill 文档
5. 写 discipline prompt，补充本学科特有禁区
6. 用一个现有人物样本试跑
7. 检查输出是否：
   - 重复
   - 越界
   - 证据不足却口气过重
   - shared constructs 映射不清
8. 再进入批量扩写

## 12. 评估一个新 skill 是否合格

可用下面这份清单快速判断：

- 是否有清楚的理论来源？
- 是否有自己的第一问题？
- 是否能从公开材料里抓到证据？
- 是否会和邻近 lens 大量重复？
- 是否能映射 shared constructs？
- 是否写清了不应越界之处？
- 是否允许输出 weak / not_applicable？
- 是否能产出 0-2 个有意义的 emergent construct？

如果这 8 个问题里有 3 个以上答不上来，这个 skill 通常还不够稳定。

## 13. 当前 medicine 的经验，也适用于其他学科

medicine 的设计过程暴露出一个对所有 discipline 都成立的事实：

> 最大的问题通常不是“知识不够多”，而是 lens 之间边界不清。

因此，新增任何领域时，优先做的都不是“多写内容”，而是：

- 收紧第一问题
- 明确优先证据
- 写清常见混淆
- 写清不应越界

理论多样性有价值，但前提是每个 lens 真的是不同的镜头，而不是同一镜头换个名字。

## 14. 一句话总结

这个项目的正确工作方式是：

> 用结构化证据约束多理论 lens 的独立分析，再通过 shared_constructs 和 critic/synthesizer 把这些分析对接起来；表达可以有趣，但底层必须可回溯、可约束、可比较。
