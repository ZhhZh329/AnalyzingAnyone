# 威胁建模分析框架 (Threat Modeling)

## 1. 核心理论框架

威胁建模要求人不只想“系统如何工作”，还想“系统如何被破坏”。用于人物分析时，它能区分一种人是默认环境善意，还是会主动推演对手、攻击路径和误用场景。

## 2. 锚点人物与流派

- **Bruce Schneier**：安全思维与威胁建模文化的重要代表。
- **Adam Shostack**：将威胁建模方法系统化推广到工程实践。
- **Ross Anderson**：从系统与激励角度理解安全机制。

## 3. 分析方法论

- 是否会主动识别攻击面与信任边界。
- 是否能具体描述对手能力，而不是抽象地说“有人会搞破坏”。
- 是否重视误用场景和滥用路径。
- 是否理解安全问题常是设计问题而非补丁问题。

## 4. Construct 分析指南

- **agency_orientation**：是否试图通过识别攻击面、对手能力、信任边界与系统脆弱点主动改写问题结构，而不是只在既有条件下被动应对。
- **epistemic_style**：其判断更依赖关于识别攻击面、对手能力、信任边界与系统脆弱点的实验、度量、模型与证据，还是经验直觉和少量轶事。
- **abstraction_formalization**：是否能把识别攻击面、对手能力、信任边界与系统脆弱点抽象成变量、接口、状态、约束、目标函数或可验证规范。
- **control_orientation**：更倾向直接操纵识别攻击面、对手能力、信任边界与系统脆弱点的每个环节，还是通过机制、反馈和边界条件实现间接控制。
- **mission_vs_relational_cost**：为了推进识别攻击面、对手能力、信任边界与系统脆弱点相关目标，是否愿意承受协作摩擦、组织阻力与人际成本。
- **risk_posture**：面对识别攻击面、对手能力、信任边界与系统脆弱点中的不确定性、失败模式与尾部风险，是激进试错、分层防御还是保守回避。
- **relation_to_hierarchy_power**：在涉及识别攻击面、对手能力、信任边界与系统脆弱点的资源、接口、权限和责任边界时，如何理解权威、分工与控制权。
- **conflict_strategy**：当围绕识别攻击面、对手能力、信任边界与系统脆弱点的定义、指标、方案或责任发生分歧时，是诉诸实验、形式化、谈判还是权力推动。
- **time_horizon**：在识别攻击面、对手能力、信任边界与系统脆弱点上更追求短期可见收益，还是愿意为长期能力建设、平台化积累与可维护性投资。
- **rebuild_over_patch**：当识别攻击面、对手能力、信任边界与系统脆弱点暴露出系统性缺陷时，是局部修补还是重写关键假设、接口和架构。
- **metacognition**：是否能反思自己在识别攻击面、对手能力、信任边界与系统脆弱点上的常见盲点，例如迷信指标、忽视边界条件、误把历史成功当普遍规律。

## 5. 典型发现模式

- 默认善意型：环境一旦变敌对，防线迅速崩塌。
- 对手敏感型：能够从攻击者视角倒推设计缺陷。
- 安全戏剧型：语言上重视安全，机制上准备不足。
- 边界清晰型：擅长把信任和权限边界设计得可执行。
