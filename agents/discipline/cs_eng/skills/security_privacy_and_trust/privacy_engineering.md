# 隐私工程分析框架 (Privacy Engineering)

## 1. 核心理论框架

隐私工程不是简单隐藏信息，而是在数据价值与主体权利之间建立可持续边界。用于人物分析时，这个框架能判断一个人是否天然尊重信息最小化、目的限制和 consent boundary。

## 2. 锚点人物与流派

- **Cynthia Dwork**：差分隐私的核心奠基者。
- **Helen Nissenbaum**：情境完整性理论的重要提出者。
- **Arvind Narayanan**：现代隐私风险与去匿名化研究的重要代表。

## 3. 分析方法论

- 是否会主动减少不必要的数据采集。
- 是否区分合法可得与伦理可用。
- 是否敏感于重识别和二次用途风险。
- 是否愿意为隐私付出效率或便利成本。

## 4. Construct 分析指南

- **agency_orientation**：是否试图通过数据最小化、用途限制、匿名化与隐私风险控制主动改写问题结构，而不是只在既有条件下被动应对。
- **epistemic_style**：其判断更依赖关于数据最小化、用途限制、匿名化与隐私风险控制的实验、度量、模型与证据，还是经验直觉和少量轶事。
- **abstraction_formalization**：是否能把数据最小化、用途限制、匿名化与隐私风险控制抽象成变量、接口、状态、约束、目标函数或可验证规范。
- **control_orientation**：更倾向直接操纵数据最小化、用途限制、匿名化与隐私风险控制的每个环节，还是通过机制、反馈和边界条件实现间接控制。
- **mission_vs_relational_cost**：为了推进数据最小化、用途限制、匿名化与隐私风险控制相关目标，是否愿意承受协作摩擦、组织阻力与人际成本。
- **risk_posture**：面对数据最小化、用途限制、匿名化与隐私风险控制中的不确定性、失败模式与尾部风险，是激进试错、分层防御还是保守回避。
- **relation_to_hierarchy_power**：在涉及数据最小化、用途限制、匿名化与隐私风险控制的资源、接口、权限和责任边界时，如何理解权威、分工与控制权。
- **conflict_strategy**：当围绕数据最小化、用途限制、匿名化与隐私风险控制的定义、指标、方案或责任发生分歧时，是诉诸实验、形式化、谈判还是权力推动。
- **time_horizon**：在数据最小化、用途限制、匿名化与隐私风险控制上更追求短期可见收益，还是愿意为长期能力建设、平台化积累与可维护性投资。
- **rebuild_over_patch**：当数据最小化、用途限制、匿名化与隐私风险控制暴露出系统性缺陷时，是局部修补还是重写关键假设、接口和架构。
- **metacognition**：是否能反思自己在数据最小化、用途限制、匿名化与隐私风险控制上的常见盲点，例如迷信指标、忽视边界条件、误把历史成功当普遍规律。

## 5. 典型发现模式

- 数据贪婪型：先收再说，后续风险高。
- 边界尊重型：清楚哪些数据不该碰、哪些用途不该越界。
- 匿名化幻觉型：低估链接攻击与重识别风险。
- 隐私工程型：擅长在产品设计中前置隐私约束。
