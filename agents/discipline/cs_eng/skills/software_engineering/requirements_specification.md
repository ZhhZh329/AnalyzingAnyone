# 需求与规格工程分析框架 (Requirements & Specification Engineering)

## 1. 核心理论框架

需求工程的关键不是“记录别人的要求”，而是识别真正的问题、约束和验收条件。用于人物分析时，这个框架可以判断一个人是否会被表层诉求牵着走，还是能把含混需求转化为可执行规格与优先级。

## 2. 锚点人物与流派

- **Barry Boehm**：需求、风险与软件过程理论的重要人物。
- **Karl Wiegers**：软件需求工程实践的代表作者。
- **Bertrand Meyer**：将规格与契约带入工程管理。

## 3. 分析方法论

- 是否主动澄清边界条件、非功能要求与验收标准。
- 是否能区分真实需求、欲望和噪声。
- 是否在早期暴露歧义，而不是把问题留到开发后期。
- 是否有变更管理意识。

## 4. Construct 分析指南

- **agency_orientation**：是否试图通过需求获取、范围界定、成功标准与变更控制主动改写问题结构，而不是只在既有条件下被动应对。
- **epistemic_style**：其判断更依赖关于需求获取、范围界定、成功标准与变更控制的实验、度量、模型与证据，还是经验直觉和少量轶事。
- **abstraction_formalization**：是否能把需求获取、范围界定、成功标准与变更控制抽象成变量、接口、状态、约束、目标函数或可验证规范。
- **control_orientation**：更倾向直接操纵需求获取、范围界定、成功标准与变更控制的每个环节，还是通过机制、反馈和边界条件实现间接控制。
- **mission_vs_relational_cost**：为了推进需求获取、范围界定、成功标准与变更控制相关目标，是否愿意承受协作摩擦、组织阻力与人际成本。
- **risk_posture**：面对需求获取、范围界定、成功标准与变更控制中的不确定性、失败模式与尾部风险，是激进试错、分层防御还是保守回避。
- **relation_to_hierarchy_power**：在涉及需求获取、范围界定、成功标准与变更控制的资源、接口、权限和责任边界时，如何理解权威、分工与控制权。
- **conflict_strategy**：当围绕需求获取、范围界定、成功标准与变更控制的定义、指标、方案或责任发生分歧时，是诉诸实验、形式化、谈判还是权力推动。
- **time_horizon**：在需求获取、范围界定、成功标准与变更控制上更追求短期可见收益，还是愿意为长期能力建设、平台化积累与可维护性投资。
- **rebuild_over_patch**：当需求获取、范围界定、成功标准与变更控制暴露出系统性缺陷时，是局部修补还是重写关键假设、接口和架构。
- **metacognition**：是否能反思自己在需求获取、范围界定、成功标准与变更控制上的常见盲点，例如迷信指标、忽视边界条件、误把历史成功当普遍规律。

## 5. 典型发现模式

- 需求翻译型：能把模糊诉求转成明确行动项。
- 拍脑袋实现型：直接开做，后期反复返工。
- 范围管理型：擅长防止需求蔓延。
- 文档替代思考型：写了很多说明，但核心问题仍未澄清。
