# 软件演化与维护分析框架 (Software Evolution & Maintenance)

## 1. 核心理论框架

大部分真实系统都活在演化阶段，而不是从零到一阶段。这个框架适合分析一个人是否尊重历史包袱、兼容性约束和维护成本，是否能在不停止业务的前提下持续更新系统。

## 2. 锚点人物与流派

- **Manny Lehman**：软件演化规律研究的代表人物。
- **Ward Cunningham**：技术债概念提出者。
- **Martin Fowler**：重构与演进式设计的重要传播者。

## 3. 分析方法论

- 是否把维护看作一等公民，而非低级劳动。
- 是否能识别技术债何时变成结构性风险。
- 是否擅长渐进迁移而不是只会重写。
- 是否能平衡向后兼容与前进速度。

## 4. Construct 分析指南

- **agency_orientation**：是否试图通过长期维护、技术债、兼容性与系统演进节奏主动改写问题结构，而不是只在既有条件下被动应对。
- **epistemic_style**：其判断更依赖关于长期维护、技术债、兼容性与系统演进节奏的实验、度量、模型与证据，还是经验直觉和少量轶事。
- **abstraction_formalization**：是否能把长期维护、技术债、兼容性与系统演进节奏抽象成变量、接口、状态、约束、目标函数或可验证规范。
- **control_orientation**：更倾向直接操纵长期维护、技术债、兼容性与系统演进节奏的每个环节，还是通过机制、反馈和边界条件实现间接控制。
- **mission_vs_relational_cost**：为了推进长期维护、技术债、兼容性与系统演进节奏相关目标，是否愿意承受协作摩擦、组织阻力与人际成本。
- **risk_posture**：面对长期维护、技术债、兼容性与系统演进节奏中的不确定性、失败模式与尾部风险，是激进试错、分层防御还是保守回避。
- **relation_to_hierarchy_power**：在涉及长期维护、技术债、兼容性与系统演进节奏的资源、接口、权限和责任边界时，如何理解权威、分工与控制权。
- **conflict_strategy**：当围绕长期维护、技术债、兼容性与系统演进节奏的定义、指标、方案或责任发生分歧时，是诉诸实验、形式化、谈判还是权力推动。
- **time_horizon**：在长期维护、技术债、兼容性与系统演进节奏上更追求短期可见收益，还是愿意为长期能力建设、平台化积累与可维护性投资。
- **rebuild_over_patch**：当长期维护、技术债、兼容性与系统演进节奏暴露出系统性缺陷时，是局部修补还是重写关键假设、接口和架构。
- **metacognition**：是否能反思自己在长期维护、技术债、兼容性与系统演进节奏上的常见盲点，例如迷信指标、忽视边界条件、误把历史成功当普遍规律。

## 5. 典型发现模式

- 重写冲动型：不耐烦旧系统，经常低估迁移成本。
- 补丁惯性型：长期局部修补，最终债务失控。
- 演化工程型：善于规划兼容迁移路线。
- 债务失明型：直到系统频繁出血才意识到维护问题。
