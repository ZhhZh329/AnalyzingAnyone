# 架构与模块化分析框架 (Architecture & Modularity)

## 1. 核心理论框架

架构不是豪华蓝图，而是对变化成本的长期管理。该框架关注一个人是否天然关心耦合、内聚、边界稳定性和演化路径，是否知道哪些地方该抽象、哪些地方该保持简单直连。

## 2. 锚点人物与流派

- **David Parnas**：模块化与信息隐藏的经典代表。
- **Grady Booch**：软件架构与复杂系统设计的重要传播者。
- **Martin Fowler**：持续演化中的架构实践与重构思想。

## 3. 分析方法论

- 是否敏感于依赖方向与模块边界。
- 是否能从组织结构、职责和沟通模式反推架构问题。
- 是否理解架构债与短期交付之间的张力。
- 是否擅长为未来变化预留接口。

## 4. Construct 分析指南

- **agency_orientation**：是否试图通过系统分层、模块边界、依赖控制与架构演化主动改写问题结构，而不是只在既有条件下被动应对。
- **epistemic_style**：其判断更依赖关于系统分层、模块边界、依赖控制与架构演化的实验、度量、模型与证据，还是经验直觉和少量轶事。
- **abstraction_formalization**：是否能把系统分层、模块边界、依赖控制与架构演化抽象成变量、接口、状态、约束、目标函数或可验证规范。
- **control_orientation**：更倾向直接操纵系统分层、模块边界、依赖控制与架构演化的每个环节，还是通过机制、反馈和边界条件实现间接控制。
- **mission_vs_relational_cost**：为了推进系统分层、模块边界、依赖控制与架构演化相关目标，是否愿意承受协作摩擦、组织阻力与人际成本。
- **risk_posture**：面对系统分层、模块边界、依赖控制与架构演化中的不确定性、失败模式与尾部风险，是激进试错、分层防御还是保守回避。
- **relation_to_hierarchy_power**：在涉及系统分层、模块边界、依赖控制与架构演化的资源、接口、权限和责任边界时，如何理解权威、分工与控制权。
- **conflict_strategy**：当围绕系统分层、模块边界、依赖控制与架构演化的定义、指标、方案或责任发生分歧时，是诉诸实验、形式化、谈判还是权力推动。
- **time_horizon**：在系统分层、模块边界、依赖控制与架构演化上更追求短期可见收益，还是愿意为长期能力建设、平台化积累与可维护性投资。
- **rebuild_over_patch**：当系统分层、模块边界、依赖控制与架构演化暴露出系统性缺陷时，是局部修补还是重写关键假设、接口和架构。
- **metacognition**：是否能反思自己在系统分层、模块边界、依赖控制与架构演化上的常见盲点，例如迷信指标、忽视边界条件、误把历史成功当普遍规律。

## 5. 典型发现模式

- 架构前瞻型：能早期识别耦合风险并建立稳定边界。
- 局部堆砌型：短期可交付，但系统复杂度持续积累。
- 抽象过度型：过早引入层次，导致执行迟缓。
- 演化稳态型：通过小步重构保持整体可控。
