# 系统集成分析框架 (Systems Integration)

## 1. 核心理论框架

集成是工程世界最容易被低估的能力之一。很多人擅长做单点模块，但无法让它们在真实世界里共同工作。用于人物分析时，这个框架能识别一个人是否拥有 interface realism、依赖管理和跨边界协同能力。

## 2. 锚点人物与流派

- **Simon Ramo**：系统工程与复杂项目集成的重要代表。
- **INCOSE 系统工程传统**：强调需求、接口、验证与集成闭环。
- **Gene Kim**：现代复杂系统交付中对跨团队流动与协同的强调。

## 3. 分析方法论

- 是否善于处理接口错配和跨团队依赖。
- 是否理解单点最优不等于整体可用。
- 是否会在早期规划集成验证。
- 是否能在多方约束中推动整体交付。

## 4. Construct 分析指南

- **agency_orientation**：是否试图通过跨模块、跨团队、跨供应链把异构组件组合成可运行整体主动改写问题结构，而不是只在既有条件下被动应对。
- **epistemic_style**：其判断更依赖关于跨模块、跨团队、跨供应链把异构组件组合成可运行整体的实验、度量、模型与证据，还是经验直觉和少量轶事。
- **abstraction_formalization**：是否能把跨模块、跨团队、跨供应链把异构组件组合成可运行整体抽象成变量、接口、状态、约束、目标函数或可验证规范。
- **control_orientation**：更倾向直接操纵跨模块、跨团队、跨供应链把异构组件组合成可运行整体的每个环节，还是通过机制、反馈和边界条件实现间接控制。
- **mission_vs_relational_cost**：为了推进跨模块、跨团队、跨供应链把异构组件组合成可运行整体相关目标，是否愿意承受协作摩擦、组织阻力与人际成本。
- **risk_posture**：面对跨模块、跨团队、跨供应链把异构组件组合成可运行整体中的不确定性、失败模式与尾部风险，是激进试错、分层防御还是保守回避。
- **relation_to_hierarchy_power**：在涉及跨模块、跨团队、跨供应链把异构组件组合成可运行整体的资源、接口、权限和责任边界时，如何理解权威、分工与控制权。
- **conflict_strategy**：当围绕跨模块、跨团队、跨供应链把异构组件组合成可运行整体的定义、指标、方案或责任发生分歧时，是诉诸实验、形式化、谈判还是权力推动。
- **time_horizon**：在跨模块、跨团队、跨供应链把异构组件组合成可运行整体上更追求短期可见收益，还是愿意为长期能力建设、平台化积累与可维护性投资。
- **rebuild_over_patch**：当跨模块、跨团队、跨供应链把异构组件组合成可运行整体暴露出系统性缺陷时，是局部修补还是重写关键假设、接口和架构。
- **metacognition**：是否能反思自己在跨模块、跨团队、跨供应链把异构组件组合成可运行整体上的常见盲点，例如迷信指标、忽视边界条件、误把历史成功当普遍规律。

## 5. 典型发现模式

- 单点英雄型：模块很强，但无法组成系统。
- 集成工程型：擅长把异构部分拼成可运行整体。
- 接口失配型：忽视边界协调，后期冲突频发。
- 整体交付型：天然关心最后一公里能否闭环。
