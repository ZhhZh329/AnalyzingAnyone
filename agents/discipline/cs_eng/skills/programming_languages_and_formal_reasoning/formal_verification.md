# 形式化验证分析框架 (Formal Verification)

## 1. 核心理论框架

形式化验证代表一种高标准思维：对关键系统，仅靠经验和测试还不够，还需要证明某些坏事不会发生。映射到人物分析时，它关注一个人是否会寻找真正不可妥协的安全边界，是否能把关键风险写成可验证条件。

## 2. 锚点人物与流派

- **Edsger Dijkstra**：程序正确性与证明文化的代表人物。
- **Amir Pnueli**：时序逻辑与系统验证的重要奠基者。
- **Leslie Lamport**：TLA+ 等形式化工具在工业系统中的推动者。

## 3. 分析方法论

- 是否区分“通常没问题”和“原则上保证不出事”。
- 是否能找出真正关键的不变量。
- 是否愿意在高风险处投入形式化成本。
- 是否知道何时验证值得做、何时成本过高。

## 4. Construct 分析指南

- **agency_orientation**：是否试图通过通过证明、模型检查与不变量验证保证关键系统行为主动改写问题结构，而不是只在既有条件下被动应对。
- **epistemic_style**：其判断更依赖关于通过证明、模型检查与不变量验证保证关键系统行为的实验、度量、模型与证据，还是经验直觉和少量轶事。
- **abstraction_formalization**：是否能把通过证明、模型检查与不变量验证保证关键系统行为抽象成变量、接口、状态、约束、目标函数或可验证规范。
- **control_orientation**：更倾向直接操纵通过证明、模型检查与不变量验证保证关键系统行为的每个环节，还是通过机制、反馈和边界条件实现间接控制。
- **mission_vs_relational_cost**：为了推进通过证明、模型检查与不变量验证保证关键系统行为相关目标，是否愿意承受协作摩擦、组织阻力与人际成本。
- **risk_posture**：面对通过证明、模型检查与不变量验证保证关键系统行为中的不确定性、失败模式与尾部风险，是激进试错、分层防御还是保守回避。
- **relation_to_hierarchy_power**：在涉及通过证明、模型检查与不变量验证保证关键系统行为的资源、接口、权限和责任边界时，如何理解权威、分工与控制权。
- **conflict_strategy**：当围绕通过证明、模型检查与不变量验证保证关键系统行为的定义、指标、方案或责任发生分歧时，是诉诸实验、形式化、谈判还是权力推动。
- **time_horizon**：在通过证明、模型检查与不变量验证保证关键系统行为上更追求短期可见收益，还是愿意为长期能力建设、平台化积累与可维护性投资。
- **rebuild_over_patch**：当通过证明、模型检查与不变量验证保证关键系统行为暴露出系统性缺陷时，是局部修补还是重写关键假设、接口和架构。
- **metacognition**：是否能反思自己在通过证明、模型检查与不变量验证保证关键系统行为上的常见盲点，例如迷信指标、忽视边界条件、误把历史成功当普遍规律。

## 5. 典型发现模式

- 证明导向型：在关键环节追求强保证，风险控制能力强。
- 测试满足型：只要样例通过就放心，容易留下系统盲区。
- 安全极小集型：善于识别少数必须被证明的核心约束。
- 形式主义过载型：验证欲望强，但与实际节奏脱节。
