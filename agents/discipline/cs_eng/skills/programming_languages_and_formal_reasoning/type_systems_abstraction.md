# 类型系统与抽象边界分析框架 (Type Systems & Abstraction Boundaries)

## 1. 核心理论框架

类型系统不仅是编程语言工具，更是一种关于边界、契约与非法状态排除的哲学。用于人物分析时，这个框架考察一个人是否会自发区分角色、接口、责任域和不变量，是否习惯在系统入口处阻断错误，而不是事后补救。

## 2. 锚点人物与流派

- **Barbara Liskov**：数据抽象与接口契约的关键人物。
- **Robin Milner**：类型理论与 ML 语言的重要奠基者。
- **Tony Hoare**：抽象、类型与程序正确性的长期倡导者。

## 3. 分析方法论

- 是否会主动划分清楚接口与实现、合法状态与非法状态。
- 是否偏好让规则在系统边界被执行，而不是靠人记住。
- 是否重视组合性与局部可推理性。
- 是否能通过命名和结构减少歧义。

## 4. Construct 分析指南

- **agency_orientation**：是否试图通过通过类型、接口与边界管理复杂性和错误传播主动改写问题结构，而不是只在既有条件下被动应对。
- **epistemic_style**：其判断更依赖关于通过类型、接口与边界管理复杂性和错误传播的实验、度量、模型与证据，还是经验直觉和少量轶事。
- **abstraction_formalization**：是否能把通过类型、接口与边界管理复杂性和错误传播抽象成变量、接口、状态、约束、目标函数或可验证规范。
- **control_orientation**：更倾向直接操纵通过类型、接口与边界管理复杂性和错误传播的每个环节，还是通过机制、反馈和边界条件实现间接控制。
- **mission_vs_relational_cost**：为了推进通过类型、接口与边界管理复杂性和错误传播相关目标，是否愿意承受协作摩擦、组织阻力与人际成本。
- **risk_posture**：面对通过类型、接口与边界管理复杂性和错误传播中的不确定性、失败模式与尾部风险，是激进试错、分层防御还是保守回避。
- **relation_to_hierarchy_power**：在涉及通过类型、接口与边界管理复杂性和错误传播的资源、接口、权限和责任边界时，如何理解权威、分工与控制权。
- **conflict_strategy**：当围绕通过类型、接口与边界管理复杂性和错误传播的定义、指标、方案或责任发生分歧时，是诉诸实验、形式化、谈判还是权力推动。
- **time_horizon**：在通过类型、接口与边界管理复杂性和错误传播上更追求短期可见收益，还是愿意为长期能力建设、平台化积累与可维护性投资。
- **rebuild_over_patch**：当通过类型、接口与边界管理复杂性和错误传播暴露出系统性缺陷时，是局部修补还是重写关键假设、接口和架构。
- **metacognition**：是否能反思自己在通过类型、接口与边界管理复杂性和错误传播上的常见盲点，例如迷信指标、忽视边界条件、误把历史成功当普遍规律。

## 5. 典型发现模式

- 边界清晰型：善于定义责任边界，降低系统沟通摩擦。
- 隐式约定型：依赖默契和经验，规模一大就失稳。
- 类型洁癖型：追求极强约束，可能牺牲迭代速度。
- 接口工程型：通过好的抽象让团队协作更顺。
