# 数据库系统设计分析框架 (Database System Design)

## 1. 核心理论框架

数据库视角关注的是信息如何被组织、约束与维护，而不是数据越多越好。它适合观察一个人是否有 schema 意识，是否理解主键、边界、依赖和事务性在现实组织中的对应物。具备这种思维的人通常不会把知识管理当作随手堆文档，而会思考结构、索引和演化成本。

## 2. 锚点人物与流派

- **Edgar F. Codd**：关系模型奠基者。
- **Jim Gray**：事务、容错与数据库工程传奇人物。
- **Michael Stonebraker**：数据库系统架构与现代数据平台的重要推动者。

## 3. 分析方法论

- 是否会主动定义数据边界、实体关系和关键约束。
- 是否在信息增长时考虑一致性与可维护性。
- 是否理解写入便利与长期查询成本之间的张力。
- 是否具备版本、历史与审计意识。

## 4. Construct 分析指南

- **agency_orientation**：是否试图通过数据建模、事务边界、一致性与信息组织方式主动改写问题结构，而不是只在既有条件下被动应对。
- **epistemic_style**：其判断更依赖关于数据建模、事务边界、一致性与信息组织方式的实验、度量、模型与证据，还是经验直觉和少量轶事。
- **abstraction_formalization**：是否能把数据建模、事务边界、一致性与信息组织方式抽象成变量、接口、状态、约束、目标函数或可验证规范。
- **control_orientation**：更倾向直接操纵数据建模、事务边界、一致性与信息组织方式的每个环节，还是通过机制、反馈和边界条件实现间接控制。
- **mission_vs_relational_cost**：为了推进数据建模、事务边界、一致性与信息组织方式相关目标，是否愿意承受协作摩擦、组织阻力与人际成本。
- **risk_posture**：面对数据建模、事务边界、一致性与信息组织方式中的不确定性、失败模式与尾部风险，是激进试错、分层防御还是保守回避。
- **relation_to_hierarchy_power**：在涉及数据建模、事务边界、一致性与信息组织方式的资源、接口、权限和责任边界时，如何理解权威、分工与控制权。
- **conflict_strategy**：当围绕数据建模、事务边界、一致性与信息组织方式的定义、指标、方案或责任发生分歧时，是诉诸实验、形式化、谈判还是权力推动。
- **time_horizon**：在数据建模、事务边界、一致性与信息组织方式上更追求短期可见收益，还是愿意为长期能力建设、平台化积累与可维护性投资。
- **rebuild_over_patch**：当数据建模、事务边界、一致性与信息组织方式暴露出系统性缺陷时，是局部修补还是重写关键假设、接口和架构。
- **metacognition**：是否能反思自己在数据建模、事务边界、一致性与信息组织方式上的常见盲点，例如迷信指标、忽视边界条件、误把历史成功当普遍规律。

## 5. 典型发现模式

- 堆料型记录者：信息多但结构混乱，后续检索成本高。
- 模式工程型：先想清 schema，再扩张内容。
- 一致性洁癖型：过度强调规范，牺牲迭代速度。
- 演化稳健型：允许局部混乱，但持续修正数据模型。
