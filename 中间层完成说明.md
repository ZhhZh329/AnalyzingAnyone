# 中间层完成说明

## 当前定位

当前中间层已经实现为一个可运行的 Gateway MVP，目标是把现有文件驱动分析工作流包装成稳定 API，而不是重写后端分析逻辑。

现有工作方式是：

1. 接收前端或联调方上传的资料包
2. 在 Gateway 内完成解压、文件解析、source 化
3. 自动生成现有工作流可读取的 `manifest.json + sources/*`
4. 创建 run，并通过后端约定的 CLI 入口触发 `main.py`
5. 读取 `run_feedback.json` 和产物文件，统一对外返回

## 已完成的能力

### 1. API 层

当前已经实现 6 个接口：

- `POST /projects`
- `POST /projects/{project_id}/ingestion-packages`
- `POST /projects/{project_id}/runs`
- `GET /projects/{project_id}/runs/{run_id}`
- `GET /projects/{project_id}/runs/{run_id}/evidence-assembly`
- `GET /projects/{project_id}/runs/{run_id}/report`

### 2. 统一对象与响应

中间层已经定义并落地：

- 项目对象 `Project`
- 资料包对象 `IngestionPackage`
- 运行对象 `Run`
- 阶段状态对象 `StageStatus`
- 报告对象 `FinalReport`
- 统一成功/失败响应骨架
- 统一错误对象

当前响应统一包含：

- `success`
- `request_id`
- `trace_id`
- `status`
- `data`
- `validation`
- `timestamp`

失败时统一返回：

- `code`
- `message`
- `details`
- `retryable`
- `stage_key`

### 3. Mixed-package ingestion

中间层现在已经支持 mixed zip 资料包，不再要求上传方自己准备好 `manifest.json`。

当前已支持的资料类型：

- `pdf`
- `docx`
- `md`
- `txt`
- `xlsx`
- `csv`

当前处理流程：

1. 保存原始 zip
2. 安全解压
3. 扫描包内文件
4. 按文件类型抽取文本
5. 把文本 materialize 成标准 `Source[]`
6. 自动生成 `input_bundle/manifest.json + sources/*`

同时会落盘中间产物：

- `package.json`
- `raw_files.json`
- `parsed_files.json`
- `sources.json`
- `warnings.json`
- `input_bundle/`

### 4. 与现有工作流适配

当前中间层没有改动 `loader.py` 的输入契约，仍然兼容：

- `manifest.json`
- `sources/*`

当前与后端对齐后的触发方式为：

```bash
uv run python main.py --input-file <run_input.json> --feedback-out <run_feedback.json>
```

其中：

- Gateway 负责生成 `run_input.json`
- workflow 负责输出 `run_feedback.json`
- Gateway 再根据 `run_feedback.json` 里的 `output_dir` 去读真实产物

后端当前稳定产物目录仍然是：

- `output/<subject_slug>/`

### 4.1 与后端 `zxy` 分支对齐情况

当前中间层已经按后端 `origin/zxy` 分支的真实入口完成适配，已对齐的内容包括：

- 工作流入口改为：
  - `uv run python main.py --input-file <run_input.json> --feedback-out <run_feedback.json>`
- Gateway 会为每次 run 自动生成：
  - `runs/{run_id}/meta/run_input.json`
  - `runs/{run_id}/meta/run_feedback.json`
- workflow 结束后，Gateway 会读取 `run_feedback.json` 中的：
  - `status`
  - `output_dir`
  - `report_path`
- 中间层读取产物时，不再假设固定 run 级输出目录，而是根据 `feedback.json` 返回的 `output/<subject_slug>/` 路径定位：
  - `assembly.json`
  - `critic_output.json`
  - `synthesis.json`
  - `report.md`

这意味着当前中间层和后端的 CLI 契约已经能接上。

### 5. 运行状态与产物读取

当前中间层已经支持：

- 创建 run
- 查询 run 总状态
- 返回阶段状态数组
- 读取 `assembly.json`
- 读取 `report.md`

当前阶段枚举按现有工作流定义为：

- `input_normalize`
- `assemble`
- `discipline`
- `critique`
- `synthesize`
- `report`

说明：

- 这周运行中的状态主要还是由 Gateway 侧维护
- workflow 当前稳定提供的是任务结束后的 `feedback.json`
- 因此 run 级轮询可用，但不是 workflow 原生持续推送每个 stage 的细粒度进度

## 本地已验证通过的内容

已经做过本地烟测并通过的场景：

- `txt + md + csv + xlsx` 混合 zip
- `pdf + docx + txt + xlsx` 混合 zip
- 包内含不支持类型 `.doc` 时，能记录 warning 而不是直接崩溃
- 包内全是不支持类型时，会明确返回 `SOURCE_MATERIALIZATION_FAILED`
- `ingestion-package -> create run -> get run -> get report` 这条链可跑通

另外已经额外验证过两类“后端对齐”场景：

- 使用后端 `zxy` 分支的真实 `--check-only --input-file --feedback-out` 入口，对 `data/elon_musk` 经由中间层生成的 `input_bundle` 做检查，结果通过
- 使用按后端 CLI 契约编写的 mock backend，验证 `run_input.json -> run_feedback.json -> output/<subject_slug>` 这条链，中间层能正确读取并返回产物

说明：

- 这证明“zip 解压/生成 bundle/后端接受输入”已经打通
- 如果要跑真实完整 LLM 流程，还需要本地补好 `llm.yaml` 或可用的 API key

## 当前限制

当前仍然有这些限制：

- 不做 OCR
- 不支持 `.doc`
- 扫描版 PDF 或图片型 PDF 可能抽不到文本
- 不支持图片、PPT 等更多格式
- 还没有做 `color_profile`
- 还没有做 debug 接口
- 还没有做完整 source CRUD
- 还没有做异步 package processing

## 还需要和后端最终对齐的点

虽然中间层主体已经可用，但正式联调前仍建议继续和后端确认：

- `feedback.json` 字段后续是否会再扩展
- workflow 是否会补持续刷新的运行态状态文件
- 同 subject 多次运行时产物覆盖问题，后端是否后续改成 run 级输出目录
- 错误对象字段是否还要扩展
- 真实模型配置（`llm.yaml` / API key）由谁提供、如何管理

## 结论

到目前为止，中间层已经完成了 MVP 主体。

可以把它理解成：

“现有工作流外面已经包上了一层稳定 Gateway，能够接收 mixed zip 资料包，自动整理成工作流输入，触发分析流程，并统一返回运行状态和最终产物。”
